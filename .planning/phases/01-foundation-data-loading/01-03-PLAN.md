  <action>Implements DATA-01 and decisions D-05 through D-15. Rewrite `app.js` (overwriting the Plan 2 stub) as a strict-mode IIFE that does the following, in order:

1. Module state: `let allBgs = []; let globalMin = 0; let globalMax = 0;` (D-15). No other state in Phase 1 — `filtered`, `sortKey`, `openId` are Phase 2+ scope and must not appear here.

2. `load()` function:
   a. Capture `const skeletonTimer = setTimeout(renderSkeleton, 150)` (D-06: 150ms-delayed skeleton — if fetch resolves before the timer fires, cancel so no flicker on fast loads).
   b. Call `fetch('data/backgrounds.json')`, then `.then(r => { if (!r.ok) throw new Error('HTTP ' + r.status); return r.json(); })`.
   c. On success: `clearTimeout(skeletonTimer)`, store data, compute `globalMin` and `globalMax` across all 67 `attributes.*.average` values, then `renderList(allBgs)`.
   d. On error: `clearTimeout(skeletonTimer)`, call `console.error('[backgrounds] load failed', { url: 'data/backgrounds.json', status: (err.response && err.response.status) || 'unknown', error: err })` per D-08, then `renderError()`.

3. `renderSkeleton()` (D-05): clear `#list`, append ~8 grey placeholder `<article class="bg-item bg-skeleton">` nodes whose DOM shape mirrors a real row (so the skeleton matches `.bg-row` shape). Use the `.bg-skeleton` / skeleton-related classes that already exist in styles.css (from Plan 2's CSS port — if none exist, add placeholder children matching `.bg-icon`/`.bg-name`/`.bg-spark`/`.bg-wage`/`.bg-chev` and rely on the existing selectors).

4. `renderError()` (D-07): clear `#list`, append a centered `<div class="error-state">` containing the text `Couldn't load backgrounds` and a `<button class="retry-btn">Retry</button>`. Attach a click listener on the button that calls `load()` again. Button should pick up amber styling from the existing `.retry-btn` / `.pill--amber` / whatever amber-accent class exists in styles.css — if no such class exists, add inline `background-color: var(--amber)` on the button as a minimum.

5. `computeGlobalMinMax(list)` (D-15): iterate every bg's every attribute's `average`, track min/max. Return `{ min, max }`. Store into module state even though nothing in Phase 1 reads them — Phase 2 sparklines need them.

6. `renderList(list)`: clear `#list`, then for each bg call `buildRow(bg)` and append to `#list`.

7. `buildRow(bg)` (D-13, D-14): build the exact DOM shape shown in the `<interfaces>` block — an `<article class="bg-item">` with `data-id="{bg.id}"` containing a `<div class="bg-row">` with the five child slots (`.bg-icon`, `.bg-name`, `.bg-spark`, `.bg-wage`, `.bg-chev`). Populate only `.bg-name` with `bg.name`. Leave the other four slots empty in Phase 1 — Phase 2 fills them. Attach a click listener on `.bg-row` (or the `.bg-item`) that calls `console.log('[click]', bg.id)` (D-14 — plumbing-validation log, not real accordion yet).

8. Icon fallback wiring (D-11): since Phase 1 does not yet render `<img>` tags, expose a helper `function wireIconFallback(imgEl) { imgEl.addEventListener('error', () => { imgEl.replaceWith(document.createTextNode('\u2694\uFE0F')); }); }` at module scope. Phase 2 will call it — the helper must exist now (D-11: "Phase 2 just flips the switch"). Icon path prefix helper `iconUrl(bg) { return 'assets/' + bg.icon; }` (D-10) must also exist now even though Phase 1 doesn't render icons.

9. Startup: wait for `DOMContentLoaded` (or rely on `defer` from the script tag), then call `load()`.

Do NOT use `innerHTML` for data content (mockup convention per CODEBASE.md §2). Use `document.createElement` for row construction. `textContent =` is fine for text. Do NOT implement: search, sort, sparkline rendering, wage badge content, icon rendering, accordion open/close — all Phase 2/3.</action>
