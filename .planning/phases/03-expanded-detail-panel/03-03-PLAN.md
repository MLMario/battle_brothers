<objective>
Fix the scroll defect reported in UAT item 5 of `03-VERIFICATION.md`: the production page is not scrollable at all, and `item.scrollIntoView(...)` has no observable effect when a row near the bottom of the viewport is opened.

Root cause (diagnosed from `index.html` + `styles.css`):
- `index.html` contains `<div id="app"><main id="list" role="list">...</main></div>` — a two-level structure with NO `#list-wrap` element.
- `styles.css` declares `#app { height: 100dvh; overflow: hidden; display: flex; flex-direction: column; }` (lines 32-40) and then declares a scroll container on `#list-wrap { flex: 1; overflow-y: auto; ... }` (lines 151-157) — but that element does not exist in the DOM.
- `#list` has no scroll styling, so it cannot scroll. `#app` is clipped to 100dvh with `overflow: hidden`. Result: any content taller than the viewport is clipped and neither the page nor any container scrolls. `scrollIntoView` walks up from `.bg-item`, finds no scrollable ancestor, and silently does nothing.

Fix strategy: make `#list` itself the scroll container by applying the same rules currently targeted at the non-existent `#list-wrap` (rename/retarget selector — no HTML changes required, no JS changes required). Keep the `--nav-h` padding-bottom so future Phase 4 bottom nav will not overlap content.

Purpose: Close the Phase 3 gap (`## Gaps for Replanning` → UAT item 5 failure), restore scrollability, and make the existing D-03 `scrollIntoView` call observable without re-writing any app.js logic.

Output: `styles.css` with `#list` styled as the flex-child scroll container. No changes to `app.js`, `index.html`, or any other CSS rule.
</objective>
